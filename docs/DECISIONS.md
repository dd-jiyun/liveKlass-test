# 파트 1. 도메인 고민 — 사용자 여정 분석

## 1-1. 강의(Klass) — 상태 전이와 정책

### 정원이 차도 강의는 닫지 않는다

수강 인원이 최대 정원에 도달해도 강의는 `OPEN` 상태를 유지합니다. 신청 시도만 409로 거부합니다.
대기열은 "자리가 생겼을 때 신청 기회를 보존"하는 장치입니다. 정원 초과를 강의 마감 트리거로 두면 대기열 설계 전체가 의미를 잃게 됩니다.

### 강의가 닫히는 조건은 두 가지뿐이다

- **자동**: 신청 마감일 경과 (스케줄러 1분 주기)
- **수동**: 크리에이터가 직접 마감

신청 마감일은 "더 이상 신청을 받지 않겠다"는 명확한 시점이므로 자동 처리가 자연스럽습니다. 수동 마감은 강의 취소나 긴급 상황을 위한 안전장치로 남겨두었습니다.

### 재오픈 흐름은 CLOSED → DRAFT → OPEN이다

CLOSED 상태에서 바로 OPEN으로 전환하지 않고 DRAFT를 경유합니다.
재오픈의 목적은 마감일을 연장하고 다시 모집하는 것입니다. 마감일 수정은 DRAFT 상태에서만 허용하므로, "초안으로 되돌리기(`reopen`) → 마감일 수정 → 오픈(`open`)"의 3단계가 의도를 가장 명확하게 표현한다고 판단했습니다.
수동 마감 시 WAITING/NOTIFIED 대기자 전원을 자동으로 CANCELLED 처리합니다. 재오픈 후에는 기존 순번을 이어받는 것보다 선착순으로 새로 모집하는 것이 더 공평하다고 생각했습니다.

### OPEN 이후 핵심 필드는 변경할 수 없다

수강료, 정원, 시작일, 종료일은 강의가 열린 뒤 변경할 수 없습니다. 이미 신청한 수강생이 인지하지 못한 조건에서 결제하거나, 정원을 줄여 현재 수강 인원이 새 정원을 초과하는 상황을 막기 위해서입니다.
신청 마감일만 예외입니다. 재오픈 시 마감일을 연장해야 하므로 DRAFT 상태에서 수정할 수 있도록 허용했습니다.


## 1-2. 수강 신청(Enrollment) — 임시 예약과 결제 확정

### PENDING(임시 예약) 20분의 의미

수강 신청이 성공하면 즉시 `PENDING` 상태가 되고 수강 인원이 1 증가합니다. 만료 시간은 20분입니다.
기차표, 수능 원서 등 업계 표준이 15~20분입니다. 20분은 "결제를 고민할 충분한 시간"과 "자리가 오래 묶이지 않는 균형"을 동시에 만족하며, 대기자 알림 수락 시간과 통일해 시스템 일관성도 높였습니다.

#### [검토한 대안]
- **10분**: 결제를 서두르게 만들어 사용성이 나빠집니다.
- **30~60분**: 자리가 너무 오래 묶여 대기자 경험이 나빠집니다.

### PENDING 상태에서도 즉시 취소할 수 있다

20분 만료를 기다리지 않고 수강생이 직접 PENDING을 취소할 수 있습니다. 취소 즉시 자리가 반환되고 대기자에게 알림이 갑니다.
신청을 잘못하거나 마음이 바뀐 수강생이 20분간 자리를 점유하면 대기자 입장에서도 손해입니다. 즉시 반환이 사용성과 대기자 경험 모두에 이롭다고 판단했습니다.

### 무료 강의는 신청 즉시 자동 확정된다

수강료가 0원인 강의에 신청하면 `PENDING` 생성 직후 시스템이 자동으로 `CONFIRMED`로 전환합니다.

#### [검토한 대안]
- **PENDING 단계 자체를 생략하고 즉시 CONFIRMED**: 수강 인원 증가 시점이 불분명해져 흐름 일관성이 깨집니다.
- **무료도 수강생이 직접 확정 요청**: 결제할 금액이 없는데 별도 행동을 요구하는 것은 UX상 어색합니다.
- **PENDING 생성 후 시스템 자동 확정(채택)**: 유료 강의와 같은 흐름을 유지하면서 이력도 남길 수 있습니다.

### 취소 가능 기간은 결제일이 아닌 강의 시작일 기준이다

수강 확정 후 취소 가능 여부는 "강의 시작 N일 전까지"로 판단합니다. 기본값은 7일이며 크리에이터가 강의마다 다르게 설정할 수 있습니다.

#### [검토한 대안]
- **결제 확정일 기준 N일 이내**: 결제를 늦게 할수록 강의 시작 이후에도 취소가 허용되는 구조적 모순이 생깁니다.
- **강의 시작일 기준 N일 전까지(채택)**: 언제 결제했든 동일한 기준이 적용됩니다. "강의 7일 전까지 취소 가능"이라는 표현이 수강생 입장에서도 직관적입니다.

### 중복 신청은 현재 활성 상태만 거부한다

`PENDING` 또는 `CONFIRMED` 상태의 신청이 이미 있으면 재신청을 거부합니다. `CANCELLED` 이력이 있는 경우는 재신청을 허용합니다.

현재 활성 상태의 중복이 실제 문제이지, 과거 이력 자체가 문제는 아닙니다. 만료 후 재신청, 재수강, 재오픈 후 신청 등 취소 이후 다시 신청하는 경우는 모두 정상 흐름입니다.


## 1-3. 대기열(Waitlist) — 순번 보장과 알림 수락

### 대기 등록은 명시적 요청으로만 가능하다

정원 초과 거부(409) 시 자동으로 대기를 등록하지 않습니다. 수강생이 직접 대기 등록 API를 호출해야 합니다.
자동 등록은 수강생이 원하지 않는 상황에서 대기열에 올라가는 혼란을 유발할 수 있습니다. 거부 응답에 "대기 등록 가능" 안내를 포함해 수강생이 직접 선택할 수 있도록 했습니다.

### 알림 받은 대기자는 일반 신청보다 우선한다

자리가 생겨 1순위 대기자에게 `NOTIFIED` 알림을 보낸 뒤 20분 수락 시간 동안, 일반 수강 신청은 409(`ENROLLMENT_WAITLIST_PRIORITY`)로 차단합니다.
대기열의 핵심 가치는 순번 보장입니다. "알림은 받았지만 자리는 아무나 가져갈 수 있다"는 구조는 사용자 신뢰를 잃습니다.

### 대기 수락 시간은 20분이다

자리가 생기면 1순위 대기자에게 20분의 수락 기회를 드립니다. 응답하지 않으면 EXPIRED 처리 후 다음 순번에게 넘어갑니다.

#### [검토한 대안]
- **자동 배정(본인 의사 없이 PENDING 생성)**: 수강생이 모르는 사이에 20분 타이머가 시작되는 것은 UX가 나쁩니다.
- **알림 후 선착순 경쟁**: 순번이 있는 대기열에서 알림만 주고 선착순으로 경쟁하면 대기열의 의미가 없습니다.
- **순번 보장 + 20분 수락 시간(채택)**: 대기열의 공정성과 수강생 제어권을 모두 만족합니다. 수락 시간은 PENDING 만료 시간(20분)과 통일해 시스템 일관성도 높였습니다.

### 유료 강의 대기 수락은 PENDING을 경유한다

대기자가 수락하면 유료 강의는 `PENDING` 상태의 수강 신청이 생성되고, 이후 20분 내 결제 확정이 필요합니다. 무료 강의는 즉시 `CONFIRMED`입니다.
유료 강의에서 결제 확인 없이 수강이 확정되는 것은 비즈니스 규칙에 맞지 않아 PENDING 경유 방식을 선택했습니다. 이 경우 NOTIFIED 20분 + PENDING 20분 = 최대 40분의 이중 창이 발생하며, 이는 의도된 동작입니다.


## 1-4. 사용자 역할 — STUDENT에서 CREATOR로 승격

사용자 역할은 `STUDENT`와 `CREATOR` 두 가지입니다. 초기 가입 시 역할이 고정되는 것이 아니라, `promoteToCreator()`를 통해 STUDENT가 CREATOR로 승격할 수 있습니다.
CREATOR가 된 이후에도 다른 크리에이터의 강의를 수강할 수 있어야 합니다.

이를 위해 강의 목록 조회 API(`GET /api/klasses`)의 동작을 다음과 같이 설계했습니다.

- STUDENT: `?status` 무관하게 항상 전체 OPEN 강의만 조회
- CREATOR (`?status=OPEN`): 전체 OPEN 강의 조회 — 수강 신청 목적
- CREATOR (status 미지정 또는 DRAFT/CLOSED): 자신이 만든 강의만 조회 — 강의 관리 목적

CREATOR가 `?status=OPEN`을 명시하면 수강 탐색 맥락으로 판단하고 전체 OPEN 강의를 반환합니다. 별도 역할 분기 없이 요청 의도를 파라미터로 표현하는 방식입니다.


## 1-5. 감사 로그(EnrollmentHistory) — 처리 주체 구분

수강 신청 상태가 바뀔 때마다 이력을 `INSERT`합니다. 수정/삭제는 불가합니다.
처리 주체를 `USER`(수강생 직접 요청)와 `SYSTEM`(무료 자동 확정, PENDING 만료 등)으로 구분해 기록합니다. "취소한 적 없다"는 분쟁 상황이나, 시스템이 자동으로 처리한 것과 수강생이 직접 요청한 것을 구분해야 할 때 추적할 수 있도록 설계했습니다.


## 1-6. 신청 내역 페이지네이션 — 미구현 결정

내 수강 목록 조회(`GET /api/enrollments/me`)에 페이지네이션을 구현하지 않았습니다.

### offset 방식이 부적합한 이유

스케줄러가 PENDING 항목을 20분마다 자동으로 삭제합니다. offset 방식은 "N번째부터 M개"를 조회하는데, 항목이 중간에 삭제되면 다음 페이지로 넘어갈 때 일부 항목이 건너뛰어집니다. 예를 들어 페이지 1에 10개가 있었는데 그 중 2개가 만료 삭제되면, 페이지 2를 조회할 때 원래 페이지 1 끝 부분에 있어야 할 항목이 사라진 위치만큼 이동해 조회에서 누락됩니다.

### cursor 기반이 기술적으로 더 적합하다

cursor 기반은 "마지막으로 본 항목의 ID(또는 시각)보다 작은/큰 것을 N개"로 조회합니다. 중간에 항목이 삭제되어도 커서 위치를 기준으로 다음 항목을 찾기 때문에 누락이 발생하지 않습니다.

### 그럼에도 구현 범위에서 제외한 이유

"내 수강 목록"은 수강생 한 명이 보유한 신청 내역이므로 데이터 수가 많지 않을 것으로 예상됩니다. cursor 기반 구현이 기술적으로는 올바른 선택이지만, 실제로 사용자가 목록을 여러 페이지에 걸쳐 탐색해야 할 만큼 데이터가 쌓이는지 불확실했습니다. 구현 비용 대비 사용자에게 체감되는 편의가 명확하지 않아 이번 범위에서는 제외했습니다.


# 파트 2. 기술적 구현 결정

## 2-1. 정원 관리 — 비관적 락 + 카운터 컬럼

수강 신청은 마지막 자리를 두고 동시에 여러 사용자가 경쟁하는 충돌이 잦은 도메인입니다.

**채택**: 강의 row에 `SELECT FOR UPDATE`를 걸고, `enrolled_count` 컬럼으로 현재 수강 인원을 관리합니다.

#### [검토한 대안]
- **낙관적 락(버전 번호)**: 충돌 발생 시 트랜잭션을 재시도해야 합니다. 마지막 자리 경쟁처럼 충돌이 집중되는 상황에서는 재시도 폭주로 성능이 오히려 저하됩니다.
- **비관적 락 + 실시간 집계 쿼리**: 잠금은 걸지만 정원 확인을 매번 `COUNT(*)` 방식으로 처리하면 쿼리 비용이 높습니다.
- **비관적 락 + 카운터 컬럼(채택)**: 잠금 범위는 해당 강의 row 하나로 최소화하고, `enrolled_count` 컬럼으로 집계 없이 즉시 정원을 검증합니다. 카운터는 PENDING 생성 시 +1, 취소 시 -1, PENDING→CONFIRMED 전환 시 변화 없습니다.

대기열 순번(position) 채번도 동일한 비관적 락 범위 내에서 `MAX(position) + 1`로 처리해 순번 중복을 방지합니다.


## 2-2. 시간 제어 — Clock 빈 주입

서비스 계층에서 현재 시각이 필요한 경우 `LocalDateTime.now()` 직접 호출 대신 `java.time.Clock` 빈을 주입받아 `LocalDateTime.now(clock)`으로 사용합니다.
PENDING 만료(20분), 취소 가능 기간 검증, 대기 알림 수락 기간 등 시간에 민감한 로직이 많습니다. `LocalDateTime.now()`를 직접 호출하면 테스트에서 시각을 조작할 수 없어 이런 로직을 검증하기 어렵습니다.

#### [검토한 대안]
- **`MockedStatic<LocalDateTime>`으로 `now()` 가로채기**: 전역 정적 메서드를 가로채므로 병렬 테스트 시 상태 오염 위험이 있습니다.
- **현재 시각을 컨트롤러에서 파라미터로 전달**: 컨트롤러는 HTTP 요청/응답만 담당해야 합니다. 현재 시각은 비즈니스 로직 내부에서 결정해야 합니다.
- **도메인 단위 테스트만 작성하고 통합 레벨 시간 테스트 포기**: 핵심 비즈니스 규칙을 통합 수준에서 검증하지 못하면 실제 결함을 테스트가 잡지 못합니다.
- **`Clock` 빈 주입(채택)**: 코드 변경은 `now()` → `now(clock)` 한 줄이고, 테스트에서 시각을 자유롭게 제어할 수 있습니다.

### 테스트에서 Clock을 제어하는 두 가지 방법

| 상황 | 방식 |
|------|------|
| 테스트 전체에서 시각이 고정됨 | `Clock.fixed()` + `@TestConfiguration` |
| 같은 테스트 내에서 시각을 앞당겨야 함 | `@MockitoBean` + `fixClock(Instant)` 헬퍼 |

`@MockitoBean Clock clock` 선언 시 `instant()`만 stub하면 `getZone()`이 `null`을 반환해 `NullPointerException`이 발생합니다. `fixClock()` 헬퍼로 두 stub을 항상 함께 호출해 누락을 방지합니다.

```java
private void fixClock(Instant instant) {
    given(clock.instant()).willReturn(instant);
    given(clock.getZone()).willReturn(ZoneId.systemDefault());
}
```

`ClockConfig`에는 `@ConditionalOnMissingBean`을 추가해 테스트 컨텍스트에서 먼저 등록된 Clock 빈이 있으면 운영 빈을 등록하지 않도록 합니다.


## 2-3. 만료 처리 — 스케줄러 방식 채택

PENDING 만료와 NOTIFIED 만료 처리를 1분 주기 스케줄러(`@Scheduled(fixedDelay)`)로 처리합니다.

#### [검토한 대안]
- **요청 시점에 처리(Lazy 방식)**: 강의에 누군가 접근해야만 만료 처리가 됩니다. 대기자가 20분을 기다렸는데 아무도 강의를 건드리지 않으면 알림이 오지 않는 상황이 발생합니다.
- **스케줄러(채택)**: 사용자 행동과 무관하게 적시 처리를 보장합니다. 알림 타이밍이 사용성의 핵심인 대기열 도메인에서 스케줄러는 필수입니다.

테스트 실행 중 스케줄러가 간섭하지 않도록 `initialDelay = 60_000`을 설정했습니다.


## 2-4. 알림 — 인터페이스 추상화

실제 알림은 구현하지 않습니다. 알림 시각을 DB(`notified_at`)에 기록하는 것으로 대체하고, 로그 출력으로 mock 처리합니다.
단, 알림 발송 레이어는 `NotificationService` 인터페이스로 추상화합니다. 실제 구현체를 교체할 때 서비스 코드를 변경할 필요가 없습니다.
알림은 `TransactionSynchronization.afterCommit()` 콜백 내에서 발송합니다. 트랜잭션이 커밋되기 전에 알림을 보내면 롤백 시 대기자에게 잘못된 정보가 전달될 수 있기 때문입니다.
afterCommit에서 LAZY 연관관계에 접근하면 JPA 세션이 이미 닫혀 `LazyInitializationException`이 발생하므로, 알림 전용 조회 쿼리에 `JOIN FETCH`를 적용해 트랜잭션 내에서 연관 엔티티를 미리 초기화합니다.


## 2-5. API 설계 결정

### 상태 전이는 액션 기반 엔드포인트로 분리한다

초기에는 `PATCH /{id}` + `status` body 방식으로 설계했으나, 다음 이유로 액션 기반 엔드포인트로 전환했습니다.

- `status: "CONFIRMED"` 또는 `"CANCELLED"` 같은 문자열 분기는 SRP 위반입니다.
- 새로운 상태가 추가될 때마다 분기 로직이 늘어나 유지보수가 어렵습니다.

#### [결과]
- `POST /api/enrollments/{id}/confirm` — 결제 확정
- `POST /api/enrollments/{id}/cancel` — 취소
- `POST /api/klasses/{id}/open` / `/close` / `/reopen` — 강의 상태 전이
- `PATCH /api/klasses/{id}` — 순수 필드 수정만

### void 액션의 HTTP 상태 코드: 200 유지

반환값이 없는 액션(`confirm`, `cancel`)은 `{ "data": null }` + 200을 반환합니다.
REST 표준상 body가 없는 응답에는 `204 No Content`가 맞습니다. 다만 현재 모든 응답이 `ApiResponse<T>` 래퍼로 감싸지는 구조라, 204로 전환하면 void 엔드포인트만 응답 포맷이 달라집니다. 현재는 일관성을 위해 200으로 유지했습니다.

### URL에 버전을 포함하지 않는다

API 경로는 `/api/klasses`처럼 버전 세그먼트 없이 설계했습니다.
버전을 URL에 포함하는 것은, 여러 버전을 동시에 운영할 외부 클라이언트가 있고 하위 호환을 깨는 변경이 예정되어 있을 때 의미가 있다고 판단했습니다. 현재 이 시스템은 둘 다 해당하지 않아 생략했습니다.
`/api/` prefix는 유지했습니다. 액추에이터, 정적 리소스 등 다른 관심사와 라우팅을 분리하기 위한 네임스페이스로 의미가 있기 때문입니다.


## 2-6. 스케줄러 — 단일 서버 전제와 확장 시 고려사항

현재 `@Scheduled(fixedDelay)` 방식은 단일 서버 운영을 전제합니다. 서버를 2대 이상 운영하면 같은 PENDING 항목을 두 스케줄러가 동시에 만료 처리하려 해, `enrolledCount`가 두 번 감소하거나 대기자에게 알림이 중복 발송될 수 있습니다.
이 문제를 해결하는 일반적인 방법은 다음과 같습니다.

- **ShedLock**: 분산 환경에서 스케줄러 잠금을 DB 또는 Redis로 관리합니다. 어느 서버 하나만 특정 시간대에 해당 작업을 실행하도록 보장합니다.
- **DB 상태 기반 처리 + 낙관적 락**: 스케줄러가 만료 대상을 조회한 뒤, 상태 변경 시 `version` 체크로 중복 처리를 방지합니다.
- **메시지 큐(Kafka, RabbitMQ)**: 만료 이벤트를 큐에 발행하고 소비자 하나가 처리하는 방식으로 중복을 원천 차단합니다.

현재 구현은 단일 서버 범위에서 정상 동작하며, 스케일아웃이 필요한 시점에 위 방법 중 하나를 적용하는 것이 적절할 것 같습니다.
