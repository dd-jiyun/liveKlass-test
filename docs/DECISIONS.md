# 수강 신청 시스템 - 설계 결정 기록

> 요구사항을 분석하며 고민되는 부분을 도출하였습니다.
> 각 결정은 "왜 이렇게 했는가"를 중심으로 작성되었습니다.

---

## 1. 강의 상태 관리

### 정원이 가득 찼을 때 강의를 닫지 않는다

**결정**: 수강 인원이 정원에 도달해도 강의는 모집 중 상태를 유지합니다. 신청만 거부합니다.

대기열의 존재 이유는 자리가 생겼을 때 신청할 수 있는 기회를 보존하는 것입니다. 정원 초과를 강의 마감 조건으로 두면 대기열 설계 전체가 의미를 잃기 때문에, 모집 중 유지가 사용자 경험과 대기열 설계 양쪽에 일관된 선택이라고 판단했습니다.


### 강의가 닫히는 조건은 신청 마감일 도달과 크리에이터 수동 마감이다

**결정**: 강의 마감 조건 = 신청 마감일 도달(자동) 또는 크리에이터 수동 전환.

신청 마감일은 "더 이상 신청을 받지 않겠다"는 명확한 시점이므로 자동 처리가 적합하다고 판단했습니다.
크리에이터 수동 마감은 강의 취소나 내부 사정 등 예외 상황을 위한 안전장치로 두었습니다.


### 크리에이터가 강의를 수동 마감하면 기존 대기자는 자동 취소된다

**결정**: 크리에이터가 강의를 직접 닫으면 대기 중이거나 신청 기회를 받은 대기자 전원이 자동으로 취소됩니다.

크리에이터가 의도적으로 강의를 닫는 행위이므로 대기 순번을 보존하는 것이 무의미합니다.
재오픈 시에는 정원이 늘거나 조건이 바뀐 상태이므로 기존 순번을 그대로 가져가는 것보다 선착순으로 새로 모집하는 것이 공평하다고 판단했습니다.


### 강의를 열기 전에 신청 마감일 유효성을 반드시 확인한다

**결정**: 신청 마감일이 현재 날짜보다 이전이면 강의 오픈을 거부합니다. 재오픈 시에도 동일하게 적용합니다.

마감일이 이미 지난 강의를 열면 아무도 신청할 수 없는 강의가 노출됩니다.
시스템이 유효하지 않은 상태를 만들지 않도록 상태 전환 입구에서 차단합니다.


### 강의가 열린 이후에는 핵심 정보를 변경할 수 없다

**결정**: 수강료, 정원, 강의 시작일, 종료일은 강의가 열린 이후 변경할 수 없습니다. 신청 마감일만 예외적으로 재오픈 시 연장할 수 있습니다.

강의가 열린 후 수강료를 올리면 이미 임시 예약 중인 수강생이 인지하지 못한 조건에서 결제하게 됩니다.
정원을 줄이면 현재 수강 인원이 새 정원을 초과하는 모순 상태가 발생합니다.
이미 신청한 수강생의 조건을 보호하기 위한 불변 원칙으로 두었습니다.

## 2. 수강 신청 관리

### 신청 후 20분 동안 자리를 임시로 점유한다

**결정**: 수강 신청이 성공하면 즉시 임시 예약 상태가 되고, 현재 수강 인원이 1 증가합니다. 만료 시간은 20분입니다.

기차표, 수능 원서 등 업계 표준이 15~20분입니다.
20분은 "결제를 고민할 충분한 시간"과 "자리가 오래 묶이지 않는 균형"을 동시에 만족합니다.
자리 반환이 빠를수록 대기자 경험도 개선됩니다.

#### 검토한 대안
- **10분**: 결제를 서두르게 만들어 사용성이 나빠집니다.
- **30~60분**: 자리가 너무 오래 묶여 대기자 경험이 나빠집니다.
- **20분**: 업계 표준이자 대기자 수락 시간과 통일해 시스템 일관성을 높입니다.


### 임시 예약 상태에서도 수강생이 직접 취소할 수 있다

**결정**: 임시 예약 중인 수강생이 직접 취소하면 즉시 취소 처리되고 자리가 반환됩니다.

20분 만료를 기다리는 것은 불필요한 대기입니다.
신청을 잘못했거나 마음이 바뀐 수강생이 20분간 자리를 점유하면 대기자 입장에서도 손해라고 판단하여, 즉시 자리를 반환하게 하였습니다.


### 무료 강의는 신청 즉시 자동으로 수강 확정된다

**결정**: 수강료가 0원인 강의에 신청하면 임시 예약 생성 직후 시스템이 자동으로 수강 확정 처리합니다. 이력에는 시스템 처리로 기록됩니다.

#### 검토한 대안
- **임시 예약 단계 생략, 즉시 확정**: 임시 예약 없이 바로 확정하면 현재 수강 인원을 늘리는 시점이 불분명해져 흐름 일관성이 깨집니다.
- **무료 강의도 수강생이 직접 확정 요청**: 결제할 금액이 0원인데 별도 행동을 요구하는 것은 사용성상 어색합니다.
- **임시 예약 생성 후 시스템 자동 확정**: 임시 예약으로 자리를 점유한 뒤 즉시 자동 전환합니다. 유료 강의와 동일한 흐름을 유지하면서 이력도 남습니다.


### 취소 가능 기간은 결제일이 아닌 강의 시작일 기준으로 계산한다

**결정**: 수강 확정 후 취소 가능 여부는 "강의 시작 N일 전까지"로 판단합니다. 기본값은 7일이며, 크리에이터가 강의마다 다르게 설정할 수 있습니다.

#### 검토한 대안

- **결제 확정일 기준 N일 이내**: 결제를 늦게 할수록 강의 시작 이후에도 취소가 허용되는 구조적 모순이 생깁니다. "강의 시작 후 환불"이라는 상황 자체가 발생하면 처리 기준이 모호해집니다.
- **강의 시작일 기준 N일 전까지**: 언제 결제했든 동일한 마감이 적용됩니다. 수강생 입장에서도 "강의 7일 전까지 취소 가능"이 직관적으로 이해하기 쉽습니다.


### 현재 활성 상태인 신청이 있을 때만 중복 신청을 거부한다

**결정**: 임시 예약 또는 수강 확정 상태의 신청이 이미 있으면 재신청을 거부합니다. 취소된 이력이 있는 경우는 재신청을 허용합니다.

현재 활성 상태의 중복이 실제 문제이지, 과거 이력 자체가 문제는 아닙니다. 만료 후 재신청, 재수강, 재오픈 후 신청 등 취소 이후에도 다시 신청할 수 있는 상황이 있기 때문에 취소 이력에는 제약을 두지 않았습니다.

## 3. 정원 관리 및 동시성 제어

### 비관적 락 + 현재 수강 인원 컬럼으로 동시성을 제어한다

**결정**: 강의 데이터에 데이터베이스 잠금을 걸고, 현재 수강 인원을 별도 컬럼으로 관리합니다.

수강 신청은 마지막 자리를 두고 동시에 여러 사용자가 경쟁하는 충돌이 잦은 도메인입니다. 잠금 없이 수강 인원을 확인하면 두 요청이 동시에 "자리 있음"을 확인하고 둘 다 신청에 성공해 정원을 초과하는 상황이 발생합니다. 이를 방지하기 위해 한 요청이 처리되는 동안 다른 요청이 대기하도록 데이터베이스 잠금을 사용했습니다.

#### 검토한 대안
- **낙관적 락 (버전 번호 방식)**: 충돌 발생 시 트랜잭션을 재시도해야 합니다. 마지막 자리 경쟁처럼 충돌이 집중되는 상황에서는 재시도 폭주로 오히려 성능이 저하됩니다.
- **비관적 락 + 실시간 인원 집계**: 잠금은 걸지만 정원 확인을 매번 전체 수강 인원을 세는 방식으로 처리하면 쿼리 비용이 높아집니다. 현재 수강 인원 컬럼으로 대체해 빠른 검증이 가능합니다.
- **비관적 락 + 현재 수강 인원 컬럼**: 강의 데이터 하나에만 잠금을 걸어 범위를 최소화하고, 현재 수강 인원 컬럼으로 집계 없이 즉시 정원을 검증합니다.
                                수강 인원은 임시 예약 시 +1, 취소 시 -1이며 수강 확정 시에는 변화가 없습니다(임시 예약이 이미 자리를 점유 중이기 때문입니다).


### 신청 알림을 받은 대기자는 일반 신청보다 우선한다

**결정**: 신청 처리 흐름에서 신청 기회를 받은 대기자를 먼저 확인해 우선 통과시킵니다. 해당 대기자가 있으면 일반 신청은 거부합니다.

대기열의 핵심 가치는 순번 보장입니다. 단순 알림 후 선착순으로 경쟁하면 대기열의 의미가 없다고 판단했습니다.
"알림을 받았지만 자리는 아무나 가져갈 수 있다"는 구조는 사용자 신뢰를 잃습니다.

## 4. 대기열 관리

### 순번이 된 대기자에게 20분 수락 시간을 준다

**결정**: 자리가 생기면 1순위 대기자에게 20분 수락 기회를 부여합니다. 미응답 시 다음 순번으로 넘어갑니다.

#### 검토한 대안
- **자동 배정 (본인 의사 없이 임시 예약 생성)**: 수강생이 모르는 사이에 20분 타이머가 시작되는 것은 사용성이 나쁩니다.
- **알림 후 선착순**: 순번이 있는 대기열에서 알림만 주고 선착순으로 경쟁하면 대기열의 의미가 없습니다.
- **순번 보장 + 수락 시간**: 대기열의 공정성과 수강생 제어권을 모두 만족합니다. 수락 시간 20분은 임시 예약 만료 시간과 통일해 시스템 일관성을 높입니다.


### 임시 예약 만료와 대기자 알림은 스케줄러가 처리한다

**결정**: 주기적으로 실행되는 스케줄러(1분 간격)가 만료된 임시 예약을 처리합니다.

#### 검토한 대안
- **요청 시점에 처리**: 강의에 누군가 접근해야만 만료 처리가 됩니다. 대기자가 20분을 기다렸는데 아무도 강의를 건드리지 않으면 알림이 오지 않는 상황이 발생합니다.
- **스케줄러**: 사용자 행동과 무관하게 적시 처리를 보장합니다. 대기열이 있는 도메인에서는 알림 타이밍이 사용성의 핵심이기 때문입니다.

## 5. 감사 로그

### 수강 신청 상태 변경 이력은 추가만 가능하다

**결정**: 수강 신청 상태가 바뀔 때마다 별도 이력 데이터를 추가합니다. 수정/삭제는 불가합니다.

"취소한 적 없다"는 분쟁 발생 시 근거 자료가 필요합니다.
누가(수강생 또는 시스템), 언제, 왜 상태를 바꿨는지가 불변 기록으로 남으면 추적이 가능합니다.
현재 수강 신청 상태와 별개로 전환 이력을 보존하는 구조입니다.

## 6. 시간 처리

### 서비스 내부의 현재 시각은 Clock 빈으로 추상화한다

**결정**: 서비스 계층에서 현재 시각이 필요한 경우 `LocalDateTime.now()` 직접 호출 대신 `java.time.Clock` 빈을 주입받아 `LocalDateTime.now(clock)`으로 사용합니다.

임시 예약 만료(20분), 취소 가능 기간 검증, 대기 알림 수락 기간 등 시간 민감한 로직이 많습니다. `LocalDateTime.now()` 직접 호출은 테스트에서 시각을 조작할 수 없어 이런 로직을 테스트하기 어렵습니다.

`Clock`을 주입받으면 테스트에서 `@MockitoBean Clock clock`으로 교체해 `given(clock.instant()).willReturn(base.plus(21, MINUTES))`처럼 원하는 시점을 정확히 지정할 수 있습니다. 운영 환경에서는 `ClockConfig`에서 `Clock.systemDefaultZone()`을 빈으로 등록해 아무런 동작 차이가 없습니다.

#### 검토한 대안

- **`MockedStatic<LocalDateTime>`으로 `now()` 호출 가로채기**: 전역 정적 메서드를 가로채는 방식으로 테스트 격리가 어렵고 병렬 테스트 시 상태 오염 위험이 있습니다. 코드 수정 없이 테스트만 복잡해지는 구조라 배제했습니다.
- **현재 시각을 컨트롤러에서 파라미터로 넘기기**: 컨트롤러는 HTTP 요청/응답만 담당해야 합니다. "지금 몇 시인가"는 비즈니스 로직이 알아야 할 사항이지 컨트롤러가 결정할 사항이 아닙니다. 실제로 요청이 들어온 뒤 서비스 내부에서 여러 시각을 생성하는 경우 파라미터 전달이 맞지 않는 상황도 생깁니다.
- **도메인 단위 테스트만 작성하고 시간 관련 통합 테스트 포기**: 임시 예약 만료, 취소 가능 기간 초과 등 핵심 비즈니스 규칙을 통합 수준에서 검증하지 못하면 실제 사용 시나리오에서 결함이 발생해도 테스트가 잡지 못합니다.
- **`Clock` 빈 주입**: 코드 변경은 최소이고(`now()` → `now(clock)`), 테스트에서 시각을 자유롭게 제어할 수 있으며, Spring 공식 패턴이기도 합니다. 운영 코드에는 아무 영향이 없습니다.

### 테스트에서 Clock을 제어하는 방법은 두 가지로 나눈다

**결정**: 테스트 중 시각 변경이 필요 없으면 `Clock.fixed()` + `@TestConfiguration`을 사용하고, 같은 테스트 내에서 시각을 앞당겨야 하면 `@MockitoBean`을 사용하되 `instant()`와 `getZone()`을 항상 함께 stub하는 헬퍼로 감쌉니다.

`@MockitoBean Clock clock`만 선언하고 `instant()`만 stub하면 `getZone()`이 `null`을 반환해 `LocalDateTime.now(clock)` 호출 시 `NullPointerException`이 발생합니다. 에러 메시지가 명확하지 않아 원인을 찾기 어렵습니다.

`Clock.fixed(instant, zone)`은 `instant()`와 `getZone()` 모두 내장하고 있어 stub 누락 문제가 구조적으로 없습니다. 다만 빈이 Spring 컨텍스트에 올라간 이후에는 값을 교체할 수 없어 시각을 중간에 바꿔야 하는 테스트에는 사용할 수 없습니다.

| 상황 | 방식 |
|------|------|
| 테스트 전체에서 시각이 고정됨 | `Clock.fixed()` + `@TestConfiguration` |
| 같은 테스트 내에서 시각을 앞당겨야 함 | `@MockitoBean` + `fixClock(Instant)` 헬퍼 |

`fixClock(Instant)` 헬퍼는 두 stub을 항상 함께 호출해 누락을 방지합니다.

```java
private void fixClock(Instant instant) {
    given(clock.instant()).willReturn(instant);
    given(clock.getZone()).willReturn(ZoneId.systemDefault());
}
```

`ClockConfig`에는 `@ConditionalOnMissingBean`을 추가해, 테스트 컨텍스트에서 먼저 등록된 Clock 빈이 있으면 운영 빈을 등록하지 않도록 합니다.

## 7. 기타

### 알림 발송은 인터페이스로 추상화하고 실제 구현은 생략한다

**결정**: 실제 이메일·푸시 알림은 구현하지 않습니다. 알림 시각을 데이터베이스에 기록하는 것으로 대체합니다. 단, 알림 발송 레이어는 인터페이스로 추상화합니다.

현재 과제 핵심에 집중하고 기간 안에 완성하기 위해서 해당 기능은 생략하였습니다.
하지만 알림이 필요하다는 것은 인지하고 있어 유연하게 확장할 수 있도록 설계하였습니다.
